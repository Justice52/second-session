GitHub Actions

GitHub Actions is an automation platform built into GitHub that runs tasks in response to events in your repository. Think of it as a set of programmable triggers and runners that help with continuous integration, continuous delivery, maintenance tasks, and automation.

Actions can be single-purpose small programs you reuse, or they can be whole scripts you write inside a workflow file. Common uses include running tests, building artifacts, and deploying applications.

Example: Python CI (short) 
# .github/workflows/python-ci.yml
name: Python CI
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]
jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.10'
      - run: python -m pip install --upgrade pip
      - run: if [ -f requirements.txt ]; then pip install -r requirements.txt; fi
      - run: pytest

Workflows

A workflow is the YAML file you place under .github/workflows. It describes the events that trigger automation, and it lists jobs and steps that will run in response.

Workflows are durable and versioned with your code. You can have many workflows per repository, each serving a different purpose such as running linters, building releases, or publishing pages.

Workflows are declarative. You specify triggers and the desired sequence of work and GitHub handles provisioning runners and showing logs.

Jobs

A job groups steps that run together on the same runner. Each job runs in an isolated virtual environment and has its own workspace. Jobs run in parallel unless you declare dependencies with the needs keyword.

Jobs are where environment choices are made: the operating system, the CPU, and access to secrets. Outputs from jobs can be passed to dependent jobs.

Quick example 
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: make build

  deploy:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - run: ./deploy.sh

Project Board

The Project Board is a flexible planning space inside GitHub where you manage work using notes, issues, pull requests, and automation. It can act like a kanban board with columns for planned, in progress, and done.

Project boards help you track status across repositories. You can automate common actions so that moving an issue to a column triggers labels or updates to linked pull requests.

Pull Requests

A pull request is the GitHub mechanism for proposing changes to a repository. It shows diffs, allows review comments, and can be connected to checks and CI workflows. Merge rules can require passing checks and approvals before changes are merged.

Use pull requests to discuss design decisions, review code, and run automated tests. They are the core collaboration feature for teams using Git.

Issues

Issues track tasks, bugs, and feature requests. An issue can be assigned to people, labeled, and linked to pull requests. They are the canonical place to capture work and discussion outside of code changes.

Use issue templates to standardize bug reports and feature requests. Close issues automatically by referencing them in a pull request message with keywords like "Closes #123".

GitHub Pages

GitHub Pages is a hosting service for static websites directly from a repository. You can publish documentation, portfolios, or project pages by pushing HTML, CSS, and JavaScript into a specific branch or directory and enabling Pages in the repository settings.

Pages integrates with Actions if you want to build and publish a static site automatically after each push. The site then becomes available at a github.io domain or a custom domain you configure.

Cheat sheet

Actions are automation. Workflows are files. Jobs are grouped steps. Pull requests propose changes. Issues capture work. Project boards organize tasks. Pages serve static sites.

Tips

Use branch rules and required checks to protect main. Store secrets in repository settings, not in code. Keep workflows small and focused. Reuse community actions when appropriate.
